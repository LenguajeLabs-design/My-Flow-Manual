What Codex is best at in this app

Codex is strongest when it can read the real files, make the edits, check the results, and keep the work moving without making you micromanage every step.

Build and edit

Ask Codex to create pages, change code, adjust styling, wire interactions, and leave the result in your workspace.

Review and debug

It can inspect files, trace behavior, spot likely problems, and suggest or implement fixes.

Research inside the flow

Codex can check docs, compare approaches, and bring the answer back into the code or page you are making.

Run the full loop

The best mode is outcome-first: ask for the change, let it implement it, then refine with follow-up notes.

Prompting

The simplest way to ask well

Codex tends to do better when your request includes a target, a constraint, and a sense of what “done” looks like.

Say the outcome first.

Start with what you want changed or built. Example: “Add a dark mode toggle to the homepage” is stronger than “Can we do dark mode?”

Add the style or guardrails second.

Say things like “keep the same visual language,” “make it feel playful,” or “don’t turn this into a full redesign.”

Use follow-ups instead of giant prompts.

It is usually better to say “make the helper smaller” after seeing the page than to pack every future preference into one message.

Ask for action when you want action.

If you want actual changes, phrase the request as a direct doing task, not just a question about possibilities.

App Map

How the app tends to work in practice

You do not need to memorize everything. What matters is understanding which surface to use for which kind of work.

Chat

Use the chat to give direction, react to drafts, tighten the scope, and keep the project moving.

Workspace files

This is where the real output lives. Codex can read, write, and link directly to the files you are building together.

Outputs folder

Use it for the polished things you want to keep or publish, like finished HTML guides and deliverables.

Work folder

Use it for drafts, scratch files, notes, or experiments that help the final output but are not the final output.

In-app browser

Useful for checking live pages, verifying what a site looks like, and testing what changed after edits.

Threads

Think of a thread like a lane of work. Keep one thread focused on one project or outcome when you can.

Modes

Know which kind of ask you’re making

A lot of friction disappears once you separate planning, implementation, review, and polish.

Planning mode

Best for: “Help me think through the structure before we build.”

Implementation mode

Best for: “Go ahead and make the change.”

Review mode

Best for: “Check this for bugs, risks, or missing pieces.”

Polish mode

Best for: “Make it clearer, smaller, sharper, or more beautiful.”

Habits

What makes Codex feel genuinely useful

These are the operating habits that usually create the best rhythm with the app.

Show, then steer.

If you have a page, screenshot, file, or repo, point Codex at the real thing instead of describing it from memory.

Keep requests concrete.

“Make the keyboard helper feel less permanent” is easier to act on than “make it better.”

Let the first draft be fast.

It is usually more efficient to get a version on the page and then react to it together.

Use Codex for the boring middle.

File setup, repetitive edits, content structuring, style cleanup, and verification are exactly where it shines.

Ask for one main change at a time when refining.

Big multi-part feedback is possible, but one sharp direction often gets the cleanest revision.

Use the browser when appearance matters.

If the question is visual, having Codex inspect the rendered result is usually better than guessing from code alone.

Use follow-up passes for personality.

Once the structure works, ask for tone, visual identity, smoother motion, or stronger clarity.

Save the polished version in outputs.

This keeps the collection clean and makes it easier to publish or link later.
